title: Google Cloud Run
description: Deploy omni-model to Cloud Run in one click, then configure a production service with Firestore and Secret Manager.
---

Cloud Run is a good fit for omni-model when you want a standard HTTPS endpoint without managing
servers. The repository includes a **Run on Google Cloud** button for a working starter service and
a `Dockerfile` for a fully controlled deployment.

<Warning>
The one-click form defaults to Firestore, but it cannot create a Firestore database or grant its
runtime service account access on your behalf. If that setup is not already present, choose `memory`
for an evaluation deployment, then follow the Firestore setup below before raising the instance
limit.
</Warning>

## One-click starter

[![Run on Google Cloud](https://deploy.cloud.run/button.svg)](https://deploy.cloud.run/?git_repo=https://github.com/tiepvuvan/omni-model)

The button builds the checked-out `Dockerfile` with Cloud Build and creates a public Cloud Run
service. It does **not** fork the repository or require Docker locally.

<Steps>
<Step title="Choose a Google Cloud project and region">
Sign in to Google Cloud, choose the project that should own the service, then select a Cloud Run
region. Cloud Run enables the services it needs during the guided flow if your account is allowed to
do so.
</Step>

<Step title="Choose storage, App Check, and an upstream">
The deploy form supplies named `OMNI_*` environment variables. It has no conditional controls, so
make these choices directly in the values:

- `OMNI_STORAGE_TYPE`: leave `firestore` for shared counters, or enter `memory` for an evaluation
  without Firestore.
- `OMNI_SECURITY_FIREBASE_APPCHECK_ENABLED`: leave `true` to accept Firebase App Check tokens, or
  enter `false` to use the generated JWT fallback alone. On Cloud Run, the Firebase project number
  is discovered from GCP metadata; an App ID allowlist is optional.
- `OMNI_PROVIDERS_DEFAULT_BASE_URL` and `OMNI_PROVIDERS_DEFAULT_API_KEY`: both are editable later.
  The form defaults to the OpenAI-compatible OpenAI endpoint and permits a blank key so an
  unauthenticated compatible endpoint can be used. With the default endpoint, add an API key before
  serving model requests.

The button generates `OMNI_JWT_SECRET` for its fallback HS256 verifier. It creates a valid service
even when no upstream key is entered; update the service environment variables (and preferably use
Secret Manager) before sending model traffic. There is no YAML file or YAML environment variable.
</Step>

<Step title="Wait for the revision and verify health">
When deployment completes, open the Cloud Run service URL. `/healthz` is intentionally public and
should return `{"status":"ok"}`:

```sh
curl https://YOUR_SERVICE_URL/healthz
```

`/v1/*` is different: it requires a JWT signed by `OMNI_JWT_SECRET`. A request without one should
return a `401`, which confirms the proxy is not an open relay.
</Step>
</Steps>

### What the button creates

`app.json` is the deploy-button contract. Its default environment variables select Firestore,
enable Firebase App Check, retain a generated JWT fallback, cap Cloud Run at **one instance**, let
the service listen on Cloud Run's `PORT`, and make the service public at the network layer. The
application protects every `/v1/*` endpoint with either a valid App Check token or a JWT; only
`/healthz` is unauthenticated.

Firestore is the GCP-friendly choice for durable shared counters. If it is unavailable, changing
the form to `memory` still makes the first deployment useful, but its counters disappear when an
instance is replaced. The service remains capped at one instance until you complete the production
Firestore and IAM setup.

<Info>
The Cloud Run button passes values as service environment variables. For long-lived API keys,
replace the optional form value with a Secret Manager binding using the production path below.
</Info>

## Production deployment: Firestore + Secret Manager

This path creates a dedicated runtime service account, stores credentials in Secret Manager, and
uses Firestore transactions for exact shared counters. All omni-model settings are Cloud Run
environment variables—there is no configuration file to create, mount, or version. It assumes a
current Google Cloud CLI and a cloned copy of this repository.

### 1. Select the project and enable APIs

In Cloud Shell or an authenticated terminal, set the values for this deployment. Use a Firestore
location that suits your data-residency requirements; a Firestore database location cannot be
changed later.

```sh
export PROJECT_ID="YOUR_PROJECT_ID"
export REGION="us-central1"
export SERVICE="omni-model"
export RUNTIME_SA="omni-model@${PROJECT_ID}.iam.gserviceaccount.com"

gcloud config set project "$PROJECT_ID"
gcloud services enable \
  run.googleapis.com \
  cloudbuild.googleapis.com \
  artifactregistry.googleapis.com \
  firestore.googleapis.com \
  secretmanager.googleapis.com
```

Your deployer needs permission to create the service and use its runtime service account. The
[Cloud Run deployment roles](https://cloud.google.com/run/docs/deploying#required-roles) list the
minimum IAM roles. If an organization policy forbids public services, use the private-service model
described in [Cloud Run's authentication documentation](https://cloud.google.com/run/docs/authenticating/overview)
instead of `--allow-unauthenticated` below.

### 2. Create Firestore and the runtime identity

Create the default Firestore **Native mode** database if the project does not already have one.
Run this database command once per project; it fails safely if `(default)` already exists.

```sh
gcloud firestore databases create \
  --database="(default)" \
  --location="YOUR_FIRESTORE_LOCATION" \
  --type=firestore-native

gcloud iam service-accounts create omni-model \
  --display-name="omni-model Cloud Run"

gcloud projects add-iam-policy-binding "$PROJECT_ID" \
  --member="serviceAccount:${RUNTIME_SA}" \
  --role="roles/datastore.user"
```

`roles/datastore.user` gives the runtime service account access to the Firestore counter documents,
without making it a project editor. The Node server uses Cloud Run's Application Default Credentials
and `GOOGLE_CLOUD_PROJECT` automatically; do not add a service-account key file to the image.

### 3. Choose the runtime environment variables

The production starter below uses Firestore and a JWT with a `sub` claim, so `user` rate limits are
per caller. It uses the easy named-variable surface; ordered rate-limit rules stay in one JSON array.
These are service environment variables, not a file:

| Variable | Value |
| --- | --- |
| `OMNI_SERVER_LOG_LEVEL` | `info` |
| `OMNI_STORAGE_TYPE` | `firestore` |
| `OMNI_STORAGE_FIRESTORE_COLLECTION` | `omni_ratelimits` |
| `OMNI_SECURITY_JWT_ENABLED` | `true` |
| `OMNI_SECURITY_JWT_SECRET` | `${OMNI_JWT_SECRET}` |
| `OMNI_SECURITY_JWT_ALGORITHMS` | `["HS256"]` |
| `OMNI_PROVIDERS_DEFAULT_TYPE` | `openai` |
| `OMNI_PROVIDERS_DEFAULT_API_KEY` | `${OPENAI_API_KEY}` |
| `OMNI_RATE_LIMITS_JSON` | `[request rule, token rule]` |

Values that must be arrays or objects use JSON. For example, the algorithms value is `["HS256"]`.
Bare `true`, `false`, `null`, and JSON numbers become their corresponding types; all other values
stay strings. Wrap an ambiguous string in JSON quotes, such as `"123"`.

For a production identity provider, replace the simple JWT profile with Firebase Auth, App Check,
Supabase, Apple attestation, or your issuer's JWKS. The [security guides](/security/custom-jwt)
cover each verifier. Never remove all `security.providers`: omni-model refuses to start without an
authentication verifier.

### Use the project's Firebase App Check instead

If this Cloud Run project is also your Firebase project and its apps are enrolled in App Check,
replace the JWT variables with the following. No project-id or project-number environment variable
is needed: the Node runtime reads both from the Cloud Run metadata server before it parses the
configuration.

| Variable | Value |
| --- | --- |
| `OMNI_SECURITY_MODE` | `all` |
| `OMNI_SECURITY_FIREBASE_APPCHECK_ENABLED` | `true` |
| `OMNI_SECURITY_FIREBASE_APPCHECK_APP_ID` | `1:YOUR_PROJECT_NUMBER:ios:YOUR_FIREBASE_APP_ID` |
| `OMNI_SECURITY_FIREBASE_AUTH_ENABLED` | `true` |
| `OMNI_SECURITY_FIREBASE_AUTH_PROJECT_ID` | `${GOOGLE_CLOUD_PROJECT}` |

This requires both the raw `X-Firebase-AppCheck` token and the Firebase Auth ID token. It attests
the client app and gives each signed-in user an independent `key: user` rate limit. To accept only
App Check, omit the `firebase-auth` entry and use `mode: any`; App Check alone identifies an app,
not a user. For a Firebase project in a different GCP project, set App Check's numeric
`projectNumber` and Firebase Auth's string `projectId` explicitly. See [Firebase App Check](/security/firebase-app-check).

### 4. Create secrets and grant the runtime account access

Create environment-variable secrets for the provider key and JWT signing secret. Keep the JWT
signing secret somewhere safe if you are using the starter HS256 verifier; your token issuer needs
it to create client JWTs.

```sh
gcloud secrets create omni-openai-api-key --replication-policy=automatic
read -r -s OPENAI_API_KEY
printf "%s" "$OPENAI_API_KEY" | \
  gcloud secrets versions add omni-openai-api-key --data-file=-
unset OPENAI_API_KEY

gcloud secrets create omni-jwt-secret --replication-policy=automatic
read -r -s OMNI_JWT_SECRET
printf "%s" "$OMNI_JWT_SECRET" | \
  gcloud secrets versions add omni-jwt-secret --data-file=-
unset OMNI_JWT_SECRET

for SECRET in omni-openai-api-key omni-jwt-secret; do
  gcloud secrets add-iam-policy-binding "$SECRET" \
    --member="serviceAccount:${RUNTIME_SA}" \
    --role="roles/secretmanager.secretAccessor"
done
```

The prompt after `read -r -s` does not echo a value. Generate the JWT secret with
`openssl rand -base64 48`, then paste it. Cloud Run exposes both secrets as environment variables;
the named `OMNI_*` values in step 3 refer to them at omni-model startup.

If you selected Firebase App Check instead, omit the `omni-jwt-secret` creation, its IAM binding,
and its `--set-secrets` entry. The OpenAI key remains unchanged.

### 5. Deploy from source

From the repository root, deploy the `Dockerfile`. The included `.gcloudignore` keeps the source
upload focused on the packages the container needs.

```sh
gcloud run deploy "$SERVICE" \
  --source . \
  --region "$REGION" \
  --service-account "$RUNTIME_SA" \
  --allow-unauthenticated \
  --port 8787 \
  --memory 512Mi \
  --max-instances 20 \
  --set-env-vars 'OMNI_SERVER_LOG_LEVEL=info' \
  --set-env-vars 'OMNI_STORAGE_TYPE=firestore' \
  --set-env-vars 'OMNI_STORAGE_FIRESTORE_COLLECTION=omni_ratelimits' \
  --set-env-vars 'OMNI_SECURITY_JWT_ENABLED=true' \
  --set-env-vars 'OMNI_SECURITY_JWT_SECRET=${OMNI_JWT_SECRET}' \
  --set-env-vars 'OMNI_SECURITY_JWT_ALGORITHMS=["HS256"]' \
  --set-env-vars 'OMNI_PROVIDERS_DEFAULT_TYPE=openai' \
  --set-env-vars 'OMNI_PROVIDERS_DEFAULT_API_KEY=${OPENAI_API_KEY}' \
  --set-env-vars 'OMNI_RATE_LIMITS_JSON=[{"name":"per-user-requests","key":"user","requests":{"limit":30,"window":"1h"}},{"name":"per-user-daily-tokens","key":"user","tokens":{"limit":30000,"window":"1d"}}]' \
  --set-secrets "\
OPENAI_API_KEY=omni-openai-api-key:1,\
OMNI_JWT_SECRET=omni-jwt-secret:1"
```

Cloud Run sets `PORT`; omni-model reads it by default. `--port 8787` tells Cloud Run to set that
value to the image's exposed port. `--allow-unauthenticated` only permits the request to reach the
container; the configured JWT verifier remains responsible for API-client authentication.

<Warning>
Pin secret versions as shown. Environment-variable secrets are resolved when a new instance starts,
so blindly using `latest` can make revisions non-reproducible. An `OMNI_*` change also creates a
new revision, so deploy after changing any setting.
</Warning>

### 6. Verify the endpoint

```sh
export SERVICE_URL="$(gcloud run services describe "$SERVICE" \
  --region "$REGION" --format='value(status.url)')"

curl "$SERVICE_URL/healthz"
```

To test `/v1/chat/completions`, send a JWT from your real identity provider. For a short HS256
smoke test, use the secret you saved in step 4 and a local Node.js runtime to mint a token with a
`sub`:

```sh
export OMNI_JWT_SECRET="YOUR_SAVED_JWT_SECRET"
export TOKEN="$(node --input-type=module -e '
  import { createHmac } from "node:crypto";
  const encode = (value) => Buffer.from(JSON.stringify(value)).toString("base64url");
  const now = Math.floor(Date.now() / 1000);
  const unsigned = `${encode({ alg: "HS256", typ: "JWT" })}.${encode({ sub: "smoke-test", iat: now, exp: now + 300 })}`;
  process.stdout.write(`${unsigned}.${createHmac("sha256", process.env.OMNI_JWT_SECRET).update(unsigned).digest("base64url")}`);
')"

curl "$SERVICE_URL/v1/chat/completions" \
  -H "authorization: Bearer $TOKEN" \
  -H "content-type: application/json" \
  -d '{"model":"gpt-4o-mini","messages":[{"role":"user","content":"Hello from Cloud Run"}]}'
```

Do not put a signing secret in a distributed client application. HS256 is a convenient starter for
server-to-server clients; mobile and browser clients should instead receive tokens from a trusted
issuer or use one of the platform attestation verifiers.

## Updating safely

Every Cloud Run configuration change creates an immutable revision. Change any setting directly as
an environment variable, then deploy the revision. For example, change the request budget to 100
per minute:

```sh
gcloud run services update "$SERVICE" \
  --region "$REGION" \
  --update-env-vars 'OMNI_RATE_LIMITS_JSON=[{"name":"per-user-requests","key":"user","requests":{"limit":30,"window":"1h"}},{"name":"per-user-daily-tokens","key":"user","tokens":{"limit":30000,"window":"1d"}}]'
```

To rotate `OPENAI_API_KEY` or `OMNI_JWT_SECRET`, add a Secret Manager version and update the
corresponding `--update-secrets` entry. To deploy code changes, rerun the source deploy command
from step 5. To roll back a bad revision, first list its name with
`gcloud run revisions list --service "$SERVICE" --region "$REGION"`, then send all traffic back to
it:

```sh
gcloud run services update-traffic "$SERVICE" \
  --region "$REGION" \
  --to-revisions YOUR_PREVIOUS_REVISION=100
```

## Published-image alternative

To skip a source build, replace `--source .` in step 5 with a release-pinned image, for example:

```sh
--image ghcr.io/tiepvuvan/omni-model:1.2.3
```

Use an actual released tag or digest rather than `latest` for reproducibility. The same service
account, secret bindings, Firestore configuration, and `OMNI_*` environment variables apply.

## Troubleshooting

| Symptom | Likely cause and fix |
| --- | --- |
| Revision never becomes ready | Read startup logs with `gcloud run services logs read "$SERVICE" --region "$REGION" --limit 50`. A missing `OMNI_*` value, missing `${VAR}`, or an empty `security.providers` is reported before the server binds. |
| Firestore `PERMISSION_DENIED` | Verify that the **runtime** service account, not only the deployer, has `roles/datastore.user` and that the Firestore database exists. |
| `401` from `/v1/*` | Expected without a JWT. Confirm the token is HS256, is signed by `OMNI_JWT_SECRET`, and includes a non-empty `sub` claim. |
| Rate limits reset or differ by instance | The service is still using `memory` storage or has more than one instance. Set `OMNI_STORAGE_TYPE=firestore` and the Firestore variables above. |
| `403` before omni-model responds | Cloud Run IAM rejected the request. Either grant the caller Cloud Run Invoker for a private service or make the service public and let omni-model's verifier authenticate API clients. |
