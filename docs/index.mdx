omni-model is an **OpenAI-compatible AI proxy you run yourself**. Your provider API keys stay on
your infrastructure — never inside an app binary. Clients authenticate with what they already have
(Firebase App Check, Apple App Attest / DeviceCheck, Firebase Auth, Supabase, or any JWT).
Environment variables configure rate limits (request windows **and** token budgets) plus
CEL-expression model routing across OpenAI, Anthropic, Google Gemini, and any OpenAI-compatible
endpoint.

## Two ways to run it

  <Card title="Serverless (Firebase)" icon="fire" href="/installation/firebase">

## How it works

```text
Client (any OpenAI SDK, or Firebase callable)
        │  { "model": "smart", "messages": [...] }
        ▼
┌─────────────────────────────────────────────┐
│  omni-model — your infrastructure           │
│   1. authenticate   App Check / App Attest / │
│                     Firebase Auth / JWT      │
│   2. rate limit     request windows +        │
│                     token budgets            │
│   3. route          CEL rules over model,    │
│                     user claims, headers     │
└───────────────┬─────────────────────────────┘
                │  translated on the fly
      ┌─────────┼───────────────┬───────────────────────┐
      ▼         ▼               ▼                       ▼
   OpenAI   Anthropic   Google Gemini   any OpenAI-compatible endpoint
```

Requests to Anthropic and Gemini are translated to and from the OpenAI wire format automatically —
streaming included — so your client keeps speaking one protocol everywhere.

  </Card>
  <Card title="CEL model routing" icon="route" href="/model-routing/routing">
    Map a client-facing alias like <code>"smart"</code> to a concrete provider + model by user
    tier, request shape, or headers.
  </Card>
  <Card title="Native client SDKs" icon="mobile" href="/integrations/swift">
    Drop-in usage from Swift, Kotlin, React Native, Flutter, and the web.
