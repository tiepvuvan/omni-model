---
title: Firebase App Check
description: Verify that requests to the self-hosted proxy come from an authentic instance of your app.
---

Firebase App Check attests that a request originates from an authentic,
untampered instance of your app rather than an automated script or a stolen key.
The `firebase-app-check` verifier validates the App Check token that the
Firebase SDK attaches, on the **self-hosted HTTP proxy**.

App Check tokens carry no user identity — they attest the app *instance*. The
Firebase app id (`sub`) becomes `device.id`, and `user.id` stays unset. Pair
this verifier with a user token verifier such as
[`firebase-auth`](/security/custom-jwt) when you need both.

<Note>
**Self-hosted vs. serverless.** On the self-hosted proxy you enforce App Check
with this verifier. The serverless [Firebase Extension](/installation/firebase)
enforces App Check **natively** on its callable functions — the client SDK
attaches the token automatically and you do not configure a verifier there.
</Note>

## Configuration

```sh
export OMNI_SECURITY_JSON='{
  "mode":"any",
  "providers":[{
    "type":"firebase-app-check",
    "projectNumber":"1234567890",
    "appIds":["1:1234567890:ios:abcdef123456"]
  }]
}'
```

<ParamField path="projectNumber" type="string">
The **numeric** Firebase project number — not the project id. The verifier pins
the issuer to `https://firebaseappcheck.googleapis.com/<projectNumber>` and the
audience to `projects/<projectNumber>`. It is optional for `@omni-model/node`
on Cloud Run or Compute Engine: startup reads the numeric project id from the
Google metadata server. Specify it for local/container deployments or whenever
the Firebase project differs from the GCP project that runs the proxy.
</ParamField>
<ParamField path="appIds" type="string[]">
Optional allowlist of Firebase app ids, matched against the token `sub`. When
omitted, any app in the project is accepted.
</ParamField>
<ParamField path="header" type="string" default="x-firebase-appcheck">
Header carrying the raw App Check token (no `Bearer` prefix).
</ParamField>
<ParamField path="consume" type="boolean" default="false">
Enable Firebase's limited-use token replay protection. The Node runtime consumes each token using
the Firebase Admin SDK; a second use returns `401`. Only enable this after every client obtains a
new limited-use token for every proxy request. It is supported by `@omni-model/node` on Cloud Run,
GCE, or another Node environment with Application Default Credentials; Workers do not support it.
</ParamField>
<ParamField path="clockToleranceSeconds" type="integer" default="60">
Allowed clock skew, in seconds.
</ParamField>

Tokens are RS256 JWTs verified against Google's App Check JWKS
(`https://firebaseappcheck.googleapis.com/v1/jwks`). A token with no subject, or
a subject outside the `appIds` allowlist, is rejected with a token-free reason.

## Replay protection with limited-use tokens

Session App Check tokens are intentionally reusable until their expiry. For low-volume, costly, or
security-sensitive proxy operations, set `consume: true` (or the named environment variable) to
make Firebase consume each token after its local signature, project, and app-id checks pass:

```sh
export OMNI_SECURITY_FIREBASE_APPCHECK_CONSUME=true
```

This adds a Firebase verification round trip to every request and fails closed if it cannot be
completed. The Cloud Run runtime service account needs the Firebase App Check Token Verifier role:

```sh
gcloud projects add-iam-policy-binding "$PROJECT_ID" \
  --member="serviceAccount:${RUNTIME_SA}" \
  --role="roles/firebaseappcheck.tokenVerifier"
```

Clients must request a **new limited-use token per request**; an ordinary cached App Check session
token will succeed once and then be rejected as a replay. For example, on Apple platforms use
`try await AppCheck.appCheck().limitedUseToken()` instead of `token(forcingRefresh:)` before
placing the result in `X-Firebase-AppCheck`. For Firebase callable functions, enable their
`CONSUME_APP_CHECK_TOKEN` install parameter and configure the callable client for limited-use
tokens. Firebase documents the latency and attestation-quota tradeoff for this beta protection in
its [custom-backend replay-protection guide](https://firebase.google.com/docs/app-check/custom-resource-backend).

<Note>
**GCP zero-config.** When the Node server runs on Cloud Run or Compute Engine
and the Firebase project is the same GCP project, this is enough:

```sh
export OMNI_SECURITY_PROVIDERS_JSON='[{"type":"firebase-app-check"}]'
```

The runtime obtains the project number before it starts serving requests. This
does not enroll an app in Firebase App Check or configure an attestation
provider; complete those steps in the Firebase console first. Explicit
`projectNumber` always wins, which is how cross-project deployments work.
</Note>

## Sending the token

Fetch a token with the Firebase App Check SDK on the client, then send it in the
`X-Firebase-AppCheck` header on every request. Unlike the JWT verifiers, this
header carries the **raw token** with no `Bearer` scheme.

```bash
curl https://your-proxy.example.com/v1/chat/completions \
  -H "X-Firebase-AppCheck: $APP_CHECK_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"model":"gpt-4o-mini","messages":[{"role":"user","content":"hi"}]}'
```

## Requiring App Check and a user

Use `mode: all` to require both an authentic app instance and a signed-in user
on the same request:

```sh
export OMNI_SECURITY_JSON='{
  "mode":"all",
  "providers":[
    {"type":"firebase-app-check"},
    {"type":"firebase-auth","projectId":"${GOOGLE_CLOUD_PROJECT}"}
  ]
}'
```

With `mode: all`, a missing credential fails with `credential missing for
<name>`, and the merged identity carries both `device.id` (from App Check) and
`user.id` (from Firebase Auth) into [routing and rate-limit rules](/model-routing/routing).
On Cloud Run / GCE, both values are discovered from the metadata server. Outside
GCP, use explicit `projectNumber` and `projectId` values instead.
